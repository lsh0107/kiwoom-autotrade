## CRITICAL (반드시 수정)

### C1. IP 화이트리스트 미고려
키움 REST API는 허용된 IP에서만 요청 허용. Oracle VM IP 등록 필요.
VM 회수 시 IP 변경 → 봇 중단 위험.

### C2. 주문 상태 관리 부재
- 부분 체결(partial fill) 처리 없음 → `filled_quantity`, `average_fill_price` 필드 추가 필요
- 주문 상태 머신 없음 (PENDING→FILLED 전이 규칙)
- 프로세스 크래시 시 "유령 주문" 발생 가능 → 시작 시 주문 동기화 필요

### C3. 긴급 정지(Kill Switch) 없음
- 글로벌 거래 중지 버튼
- 일일 손실 한도 초과 시 자동 중지
- 전략별 포지션 한도
- 폭주 감지 (1초에 100건 주문 등)

### C4. kiwoom-restful 라이브러리 리스크
v0.2.7, 15 stars, 비공식. 추상화 레이어로 감싸서 교체 가능하게 설계 필요.

### C5. 회원가입 엔드포인트 위험
1인 시스템에 오픈 회원가입 = 누구나 내 API로 거래 가능.
→ 회원가입 제거, 관리자 1인만 하드코딩 또는 초대 코드 방식.

## WARNING (프로덕션 전 해결)

### W1. Oracle VM 유휴 회수
장 마감 후 17.5시간 + 주말은 CPU 20% 미만 → 회수 위험.
PAYG 전환 + keepalive cron 필요.

### W2. Rate Limit 재귀 무한루프 버그
429 응답 시 재귀 호출 → RecursionError. for 루프 + max_retries로 수정.

### W3. DB 백업 전략 없음
Docker volume 단일 장애점. pg_dump → Oracle Object Storage 자동 백업 필요.

### W4. WebSocket 재연결 전략 미상세
연결 끊김 감지, 구독 복원, 프론트엔드 경고 표시 필요.

### W5. APScheduler + 4 workers = 중복 실행
4개 워커에서 같은 스케줄 실행 → 4배 주문. 1 worker 또는 별도 프로세스로 분리.

### W6. AES 키 저장 위치 미정
암호화 키가 같은 VM에 있으면 의미 없음. Oracle Cloud Vault(무료) 활용.

### W7. Rate Limit 수치 미검증
~20/sec는 커뮤니티 소스. 공식 확인 안 됨. 5/sec부터 시작해서 점진적 증가.

## MISSING (누락된 요소)

### 한국 주식시장 특수 규칙
- **T+2 결제**: 매도 대금 2일 후 사용 가능
- **가격제한폭**: ±30% 일일 상하한가
- **시장 구간**: 프리마켓(08-09), 정규(09-15:30), 시간외(15:40-18:00)
- **KRX 공휴일**: 한국 공휴일 캘린더 필요
- **VI (변동성 완화장치)**: 급등/급락 시 거래 일시중단
- **기업 이벤트**: 배당락, 액면분할 등 가격 변동

### 기타 누락
- 테스트 전략 (단위/통합/모의투자 졸업 기준)
- 재해 복구 런북
- 트레이딩 전략 정의
- 모니터링/알림 (서버 uptime 외 트레이딩 이벤트)
- 프론트엔드 JWT 저장 방식 (httpOnly cookie 권장)
- API 버전닝 (/api/v1/)
- 주문 중복 방지 (멱등성 키)

## SUGGESTIONS (권장 개선)

1. **BrokerClient 추상화**: 키움/한투 교체 가능한 인터페이스
2. **멱등성 키**: 주문 시 UUID로 중복 방지
3. **트레이딩 엔진 분리**: 웹서버와 별도 프로세스로
4. **structlog 구조화 로깅**: 주문별 감사 추적
5. **헬스체크 엔드포인트**: /api/health (DB, 키움 연결, 봇 상태)
6. **MVP는 SQLite 고려**: RAM 절약, 설정 불필요, 백업 간편
