### 23:xx - Rate Limits & 메시지 큐 심층 리서치
- **수행**: 키움/KIS API rate limits 상세 조사, Kafka vs Redis vs asyncio.Queue 비교
- **조사 항목**:
  - 키움 REST API 초당 제한 (공식 문서에 구체적 수치 미공개, 커뮤니티 기반 ~20/sec)
  - 키움 레거시 Open API+ cascading rate limits (KOAPY 소스 분석: 17초/90초/180초 대기)
  - KIS API: 실전 20/sec, 모의 5/sec, WebSocket 41종목/세션
  - KIS Token: 24시간 유효, 1분당 1회 재발급, 6시간 갱신주기
  - KIS WebSocket 41종목 우회: 다중 계좌로 세션 2배 확보
  - KIS 무한 연결시도 차단 정책 (2026.02.24 시행)
  - Kafka 최소 리소스: 1-8GB RAM, JVM+ZK 필요 → 완전히 overkill
  - Redis Streams: 25-128MB, 10만+/sec → 다중 프로세스 필요 시 적합
  - asyncio.Queue: 0 오버헤드, 1만+/sec → MVP에 최적
  - Python rate limiter 라이브러리 비교 (aiolimiter, pyrate-limiter 등)
  - kiwoom-restful 내장 rate limiting 자동관리 확인
- **결과 파일**: `.claude/memory/research-rate-limits-and-queues.md`
- **핵심 결론**:
  - Kafka는 이 프로젝트에 확실히 불필요 (500 msg/sec도 안 됨)
  - MVP: asyncio.Queue → 안정화: +SQLite 로깅 → 확장: Redis Streams
  - Rate limiter: aiolimiter 또는 kiwoom-restful 내장 기능 활용
  - WebSocket: exponential backoff + 구독 복원 패턴 필수
- **다음**: 기술 스택 최종 확정 및 프로젝트 구조 설계
