## 현재 미결정 사항

> 백테스트 결과 후 결정 필요. 형식: 번호, 배경, 선택지, 권장안, 트레이드오프
> **마지막 검토**: 2026-03-11

### 11. volume_ratio 최적값
- **배경**: 모멘텀 돌파 전략의 핵심 파라미터. 승률에 직접 영향
- **선택지**: 1.2~3.0 범위에서 백테스트 결과 기반 확정
- **기본값**: 1.5
- **결정 시점**: 백테스트 완료 후

### 12. 익절 방식: 고정 vs 트레일링
- **배경**: 고정 익절(+1.0%)은 단순하지만 큰 추세를 놓침. 트레일링은 복잡하지만 수익 극대화 가능
- **선택지**: A) 고정 +1.0% | B) 트레일링 스탑 | C) 혼합 (1차 익절 + 잔량 트레일링)
- **트레이드오프**: 단순성 vs 수익 극대화
- **결정 시점**: 백테스트 비교 후

### ~~13. 종목 유니버스~~ → C) 혼합 (KOSPI 20 + KOSDAQ 10)
- **확정일**: 2026-03-09
- **반영**: `scripts/screen_symbols.py` UNIVERSE dict (30종목 고정)
- **근거**: 52주 신고가 전용 API 없어 전 종목 순회 비현실적. 고정 유니버스가 API 비용/안정성 최적
- **향후**: 모의매매 결과 기반 종목 추가/제거 가능

### 14. LLM 트리거 조건 세부 정의
- **배경**: 현재 "장 전반적 분위기 확인"만 명시. 구체적 트리거 기준 필요
- **선택지**: A) 뉴스 감성 분석 | B) 거시 지표 체크 | C) 섹터 모멘텀 | D) 복합
- **트레이드오프**: 정확도 vs API 비용/지연
- **결정 시점**: 전략 v1.0 백테스트 후 v1.1에서

### 15. 텔레그램 양방향 통신 아키텍처
- **배경**: 현재 단방향(시스템→사용자) 알림만 구현. 최종 목표는 사용자가 텔레그램으로 지시 → LLM이 해석 → 전략 수정/매매 실행
- **선택지**: A) Webhook (서버에 텔레그램이 push) | B) Long Polling (봇이 주기적 확인) | C) 혼합
- **하위 결정**:
  - LLM 모델 선택 (Claude vs GPT, 비용/속도 트레이드오프)
  - 명령 범위 (파라미터 수정만 vs 즉시 매매 포함)
  - 안전장치 (확인 단계 필요 여부, 금액 제한)
- **트레이드오프**: 편의성 vs 보안 리스크 (텔레그램으로 매매 명령은 인증/확인 필수)
- **결정 시점**: Phase 2-3, WebSocket 안정화 후
- **관련 문서**: `design-telegram-bidirectional.md`
