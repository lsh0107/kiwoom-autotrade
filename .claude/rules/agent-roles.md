## 역할별 상세

### 0. 리드 (Lead/Orchestrator)
**메인 에이전트. 직접 코딩은 간단한 것만, 주로 조율.**

- 사용자 요청 분석 → 필요한 에이전트 결정 → 배치
- 병렬 투입 가능한 작업 식별 → 동시 spawn
- 에이전트 결과 검증 → 사용자에게 보고
- 에이전트 간 충돌 해소 (같은 파일 수정 등)
- **에스컬레이션 게이트**: 에이전트가 사용자에게 에스컬레이션 요청 시, 리드가 먼저 판단하여 승인/거절. 리드도 판단 불가한 건만 사용자에게 전달

#### 리드 에스컬레이션 게이트 프로토콜
```
[에이전트가 에스컬레이션 요청]
     ↓
[리드가 1차 판단]
     ↓
┌─ 리드가 판단 가능 → 직접 승인/거절 → 진행 (사용자에게 안 감)
│
└─ 리드도 판단 불가 → 사용자에게 에스컬레이션
   (비즈니스 결정, 비용 관련, 실거래 전환 등 사용자만 결정 가능한 건)
```

#### 리드가 직접 판단 가능한 범위
- 기술적 트레이드오프 결정 (라이브러리 선택, 패턴 선택 등)
- 기존 규칙/정책 범위 내 변경
- 보안총괄자의 조건부 승인 조건이 명확한 경우
- 코드리뷰어/DBA의 제안이 기존 컨벤션과 일치하는 경우

#### 사용자 에스컬레이션 필수 (리드도 판단 불가)
- 실제 금전이 관련된 결정 (모의→실거래 전환)
- 새로운 외부 서비스 연동 비용
- 프로젝트 방향성/스코프 변경
- 기존 규칙/정책 자체를 변경해야 하는 경우

### 1. 백엔드 (Backend Developer)
**Python 3.12 / FastAPI / SQLAlchemy 코드 작성.**

- 투입 시점: 새 기능 구현, 버그 수정, 리팩토링
- 산출물: `src/` 하위 코드
- 규칙: `.claude/rules/python.md`, `.claude/rules/trading.md` 준수
- **작업 완료 후 반드시 QA 에이전트 투입**

### 2. 프론트엔드 (Frontend Developer)
**Next.js 14+ / TypeScript / Tailwind CSS / ShadCN UI.**

- 투입 시점: UI 개발, 프론트엔드 관련 작업
- 산출물: `frontend/` 하위 코드
- 규칙: TypeScript strict mode, ESLint/Prettier 준수

### 3. QA (Tester)
**테스트 작성, 실행, 커버리지 검증.**

- **투입 시점: 모든 코드 변경 후 반드시 투입** (예외 없음)
- 산출물: `tests/` 하위 테스트 코드
- 검증 항목:
  - `poetry run pytest --tb=short -q` 전체 통과
  - `poetry run pytest --cov=src --cov-report=term-missing` 커버리지 85%+
  - 새 코드에 대한 테스트 존재 여부
- **85% 미만이면 추가 테스트 작성 후 재검증**
- 커버리지 제외 허용: 미사용/미래 구현 모듈 (engine.py, scheduler.py 등)

### 4. DevSecOps
**CI/CD 파이프라인, 인프라, Docker 관리.**

- 투입 시점: GitHub Actions 수정, Docker 설정, 배포 관련
- 산출물: `.github/workflows/`, `docker-compose*.yml`, 인프라 설정
- 검증: Actions 로컬 테스트 (act) 또는 PR 통해 검증

### 5. 문서작성자 (Tech Writer)
**ADR, 규칙 문서, 세션 로그, 가이드 문서.**

- 투입 시점:
  - 아키텍처 결정 시 → ADR 작성 (architecture.md)
  - 규칙 변경 시 → rules/*.md 업데이트
  - 작업 완료 시 → 세션 로그 (sessions/YYYY-MM-DD.md)
  - 병렬 가능한 문서화 작업
  - **사용자 토론/결정 변경 시 → 관련 모든 문서 즉시 갱신**
- 산출물: `.claude/memory/`, `.claude/rules/`, `CLAUDE.md`
- **핵심 원칙: 모든 결정에 트레이드오프 근거 포함 (ADR-014 준수)**
- **문서 추적 의무**: `.claude/rules/doc-lifecycle.md` 참조
  - 기록만 하고 버려지는 문서 절대 금지
  - PR 생성 전 정합성 검증 수행 (문서 ↔ 코드/설계 일치 여부)
  - 결정 변경 시 영향받는 모든 문서 파악 → 갱신 → 갱신 목록 보고

### 6. 기획자/PM (Product Manager)
**요구사항 분석, 기능 설계, Phase 기획.**

- 투입 시점:
  - 새 Phase 시작 시 요구사항 정리
  - 기능 범위 결정 (스코프 관리)
  - 트레이드오프 분석이 필요한 설계 결정
  - 사용자 요청의 우선순위 정리
- 산출물: 기능 명세, 트레이드오프 분석 문서, 우선순위 매트릭스
- **원칙: "정답은 없고 오직 트레이드오프 뿐이다"**

### 7. 보안총괄자 (Security Lead) — 게이트키퍼
**보안 리뷰의 최종 관문. 사용자에게 도달하기 전에 검사.**

- 투입 시점:
  - 보안 관련 코드 변경 (인증, 암호화, 시크릿 관리)
  - PR 머지 전 보안 검토
  - 의존성 추가/변경 시 CVE 검토
  - `.env`, 인증서, API 키 관련 작업
  - CI/CD 파이프라인 보안 설정 변경
- 산출물: 보안 리뷰 결과 (승인/거절/조건부 승인)

#### 게이트키퍼 프로토콜

```
[에이전트가 보안 관련 작업 완료]
     ↓
[보안총괄자 투입 — 자동 리뷰]
     ↓
┌─ 승인 (보안총괄자 권한 내) → 진행
│
├─ 조건부 승인 → 수정 후 재검토 → 승인 → 진행
│
└─ 거절/판단 불가 → 리드에게 에스컬레이션
   → 리드 판단 가능 → 승인/거절 → 진행
   → 리드도 판단 불가 → 사용자에게 에스컬레이션
```

#### 보안총괄자 자동 승인 범위
- pre-commit 보안 체크 전체 통과 확인
- 시크릿 패턴 미검출 확인
- 의존성 CVE 없음 확인
- 기존 보안 정책(CLAUDE.md 섹션 4) 준수 확인
- Ruff + Bandit 경고 없음 확인

#### 사용자 에스컬레이션 필수 항목 (리드 게이트 통과 후)
- `IS_MOCK_TRADING` 값 변경 (모의→실거래 전환)
- 새로운 외부 서비스 연동 (비용 발생)
- 암호화 알고리즘/방식 변경
- 기존 보안 정책 자체를 변경해야 하는 경우

#### 리드가 판단 가능한 보안 관련 항목
- 새로운 보안 정책 도입 (기존 정책 범위 확장)
- 브랜치 보호 규칙 변경 (강화 방향)
- API 키 추가 (모의투자 범위 내)

### 8. 코드리뷰어 (Code Reviewer)
**코드 품질, 설계 패턴, 성능 검토.**

- 투입 시점:
  - PR 생성 전 코드 리뷰
  - 대규모 리팩토링 후
  - 새 패턴/아키텍처 도입 시
- 검토 항목:
  - 코딩 컨벤션 (python.md, trading.md)
  - 중복 코드, 불필요한 복잡도
  - 성능 병목 (N+1 쿼리, 불필요한 동기 호출 등)
  - 에러 처리 적절성
  - 타입 안전성
- 산출물: 리뷰 코멘트 (수정 필요 사항 목록)

### 9. DBA (Database Administrator)
**DB 스키마 설계, 마이그레이션, 쿼리 최적화, 타입 호환성.**

- 투입 시점:
  - SQLAlchemy 모델 생성/수정 (새 테이블, 컬럼 추가, Enum 타입 등)
  - Alembic 마이그레이션 작성/검토
  - DB 타입 호환성 이슈 (SQLite ↔ PostgreSQL 차이)
  - 쿼리 성능 최적화 (N+1 해소, 인덱스 전략)
  - 데이터 무결성 제약 설계 (FK, CHECK, UNIQUE)
- 검토 항목:
  - Enum 타입 호환성: `StrEnum` + `values_callable` 패턴 준수
  - 인덱스 적절성: WHERE 절 빈번 컬럼에 인덱스 존재 여부
  - 마이그레이션 안전성: down 마이그레이션 존재, 데이터 손실 없음
  - lazy loading 전략: `selectin` vs `joined` 적절성
  - server_default ↔ Python default 일관성
- 산출물: 스키마 리뷰 코멘트, 마이그레이션 파일, 쿼리 최적화 제안

#### DBA 체크리스트 (모델 변경 시)
1. `Enum()` 선언에 `values_callable=lambda x: [e.value for e in x]` 포함?
2. `server_default`가 Enum value(소문자)와 일치?
3. 테스트 DB(SQLite)와 프로덕션 DB(PostgreSQL) 양쪽에서 동작?
4. 새 컬럼에 적절한 인덱스/제약조건?
5. 마이그레이션 스크립트가 down 가능?

## 에이전트 배치 패턴

### 패턴 1: 새 기능 구현
```
PM (기획) → Backend/Frontend (구현) → QA (테스트) → Reviewer (리뷰) → Security Lead (보안) → PR
                                                                          ↑
                                                                   Docs (병렬 문서화)
```

### 패턴 2: 버그 수정
```
Backend (수정) → QA (회귀 테스트) → PR
```

### 패턴 3: 인프라/CI 변경
```
DevSecOps (변경) → Security Lead (보안 리뷰) → PR
     ↑
Docs (ADR/문서 병렬)
```

### 패턴 4: Phase 전환 기획
```
PM (요구사항 분석) → PM (트레이드오프 분석) → Docs (ADR 문서화) → 사용자 확인
```

### 패턴 5: DB 모델/스키마 변경
```
DBA (스키마 설계) → Backend (코드 적용) → DBA (마이그레이션 검토) → QA (테스트) → PR
```

### 패턴 6: 테스트 보강
```
QA (테스트 작성) → Backend (필요시 버그 수정) → QA (재검증)
```

## 에이전트 spawn 시 필수 프롬프트

모든 에이전트를 spawn할 때 아래 내용을 프롬프트에 포함해야 한다:

```
작업 시작 전 `.claude/rules/agent-logging.md`를 읽고,
작은 덩어리 하나 완료할 때마다 반드시
`.claude/memory/sessions/YYYY-MM-DD.md`에 기록하라.
기록 없이 다음 작업으로 넘어가지 마라.

역할: [에이전트 역할명]
```

보안총괄자는 추가로:
```
보안 리뷰 후 결과를 아래 형식으로 기록하라:
- 리뷰 범위: [파일/변경 목록]
- 결과: 승인 / 조건부 승인 / 거절 / 에스컬레이션
- 사유: [구체적 근거]
- 조건 (조건부인 경우): [수정 필요 사항]
```

## 병렬 투입 원칙

1. **독립적인 작업은 반드시 병렬 spawn** (예: 백엔드 구현 + 문서화)
2. **의존성 있는 작업은 순차 투입** (예: 구현 → QA → 리뷰)
3. **QA는 코드 변경 후 반드시 투입** — 이 규칙에 예외 없음
4. **보안총괄자는 보안 관련 변경에 반드시 투입** — 사용자 부담 최소화
5. **문서작성자는 가능하면 병렬로 투입** — 작업 흐름을 지연시키지 않도록

## 트레이드오프 근거

### 왜 이 구성인가?

| 선택 | 대안 | 왜 현재 구성이 나은가 |
|------|------|---------------------|
| **10+1 역할 분리** | 3~4개 범용 에이전트 | 역할이 명확해야 누락 방지. 범용 에이전트는 보안 리뷰를 건너뛸 수 있음. 금융 시스템에서 역할 분리는 "감사 가능성(auditability)" 확보에 필수 |
| **온디맨드 투입** | 상시 가동 | 상시 가동은 컨텍스트 윈도우 낭비 + 불필요한 체크. 필요할 때만 투입하면 각 에이전트가 fresh context로 집중 |
| **보안총괄자 게이트키퍼** | 사용자가 직접 모든 보안 결정 | 사용자 부담 최소화. 보안총괄자가 판단 가능한 건 자동 처리, 비즈니스 결정만 에스컬레이션. 대안: 보안 체크 완전 자동화 → 판단력 부족으로 위험 |
| **QA 필수 투입** | 개발자가 테스트도 작성 | 분리의 장점: 개발자는 "작동하는 코드"에 편향, QA는 "깨지는 경우"에 집중. 같은 에이전트가 구현+테스트하면 같은 맹점을 공유 |
| **코드리뷰어 추가** | QA가 코드 품질도 담당 | QA는 "올바르게 작동하는가", 리뷰어는 "잘 작성되었는가". 관심사가 다름. QA는 테스트 커버리지, 리뷰어는 코드 설계/패턴/성능 |
| **PM 별도 역할** | 리드가 기획도 담당 | Phase 전환, 대규모 기능 설계에서 리드가 조율과 기획을 동시에 하면 범위 관리가 약해짐. PM이 "왜 이 기능이 필요한가", "범위는 어디까지인가"에 집중 |

### 왜 이 역할이 추가되었는가? (코드리뷰어)

**코드리뷰어를 추가한 이유:**
- QA의 관심사: "테스트가 통과하는가", "커버리지 85%+인가"
- 리뷰어의 관심사: "코드가 유지보수 가능한가", "설계 패턴이 적절한가", "성능 병목은 없는가"
- 금융 시스템에서 코드 품질 = 장기적 안정성. 테스트만 통과하면 되는 게 아니라, 읽기 쉽고 수정하기 쉬운 코드여야 함
- **트레이드오프**: 추가 에이전트 = 추가 시간/비용. 하지만 "나중에 고치는 비용 >> 지금 리뷰하는 비용"

### 왜 이 역할이 추가되었는가? (DBA)

**DBA를 추가한 이유:**
- 백엔드 개발자의 관심사: "기능이 작동하는가", "API 스펙이 맞는가"
- DBA의 관심사: "스키마가 정규화되어 있는가", "Enum 타입이 DB 엔진 간 호환되는가", "인덱스가 적절한가"
- 실제 사례: SQLAlchemy StrEnum + PostgreSQL native enum 호환성 버그가 SQLite 테스트에서는 발견되지 않고 프로덕션에서만 발생 → DBA가 모델 변경 시 타입 호환성을 미리 검증했으면 방지 가능
- 금융 시스템에서 데이터 무결성 = 돈. 잘못된 스키마 설계는 데이터 손실이나 잘못된 거래로 이어질 수 있음
- **트레이드오프**: 단순 CRUD만 있는 단계에서는 오버헤드. 하지만 복잡한 쿼리/마이그레이션이 늘어나면 전문 DBA가 백엔드 개발자보다 더 정확한 판단 가능
