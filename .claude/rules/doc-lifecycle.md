# 문서 생명주기 관리 규칙 (MANDATORY)

**기록만 하고 버려지는 문서는 절대 허용하지 않는다.**
**모든 문서는 추적 가능해야 한다.**

---

## 1. 문서 분류

모든 `.claude/memory/` 하위 문서는 아래 3가지 중 하나로 분류된다:

| 분류 | 설명 | 예시 | 관리 방식 |
|------|------|------|----------|
| **활성 (Active)** | 현재 진행 중인 프로젝트 상태 반영 | project.md, architecture.md, decisions-pending.md | 변경 발생 시 즉시 갱신 |
| **참조 (Reference)** | 과거 리서치/설계, 현재도 참고 가치 있음 | design-v1.1.md, research-*.md | 상단에 결정 반영 노트 필수 |
| **기록 (Archive)** | 날짜별 이력, 변경 금지 | sessions/*.md | append만, 수정/삭제 금지 |

---

## 2. 문서 헤더 규칙

**활성/참조 문서의 상단에 반드시 포함:**

```markdown
> **마지막 검토**: YYYY-MM-DD
> **상태**: [활성 | 참조 | 아카이브]
> **관련 ADR**: ADR-XXX (있으면)
```

참조 문서는 추가로:
```markdown
> **적용된 결정**: [이 리서치/설계의 결론이 어떻게 반영되었는지 명시]
```

## 3. 문서 갱신 트리거

| 이벤트 | 갱신 대상 | 담당 |
|--------|----------|------|
| 아키텍처 결정 | architecture.md + 관련 문서 | 문서작성자 |
| 코드 구현 완료 | project.md + decisions-pending.md | 문서작성자 |
| Phase 전환 | project.md + design-v1.1.md | PM + 문서작성자 |
| 새 리서치 | research-*.md 신규 생성 | 문서작성자 |
| 리서치 기반 결정 확정 | 해당 research-*.md 상단에 결정 반영 | 문서작성자 |
| 작업 완료 | sessions/YYYY-MM-DD.md | 모든 에이전트 |

## 4. 문서담당자 정기 검증 프로토콜 (MANDATORY)

### 검증 시점
- **모든 PR 생성 전** (작업 완료 시점)
- **Phase 전환 시**
- **3개 이상 ADR이 추가되었을 때**

### 검증 항목

```
1. [일관성] 모든 활성 문서가 현재 코드/설계와 일치하는가?
   - project.md의 "현재 상태"가 실제와 맞는가?
   - architecture.md의 ADR이 실제 구현과 일치하는가?
   - decisions-pending.md에 해결된 항목이 방치되어 있지 않은가?

2. [추적성] 모든 결정이 문서에 반영되어 있는가?
   - 새 ADR이 관련 문서에 교차 참조되어 있는가?
   - 코드 변경에 대응하는 문서 갱신이 있는가?

3. [참조 문서 건강도] 참조 문서의 상단 노트가 최신인가?
   - research-*.md의 "적용된 결정"이 현재 상태를 반영하는가?
   - design-v1.1.md의 변경 사항 노트가 갱신되어 있는가?

4. [누락] 문서화되지 않은 결정이 있는가?
   - 코드에 반영되었지만 ADR이 없는 주요 결정은?
   - 새로 추가된 모듈/기능에 대한 문서가 있는가?

5. [죽은 문서] 더 이상 유효하지 않은 문서가 있는가?
   - 완전히 대체된 문서 → 상단에 "대체됨: [새 문서]" 명시
   - 부분 변경된 문서 → 변경 사항 노트 갱신
```

### 검증 결과 보고 형식

```markdown
### [문서담당자] 문서 정합성 검증 결과
- **검증일**: YYYY-MM-DD
- **검증 범위**: [전체 / 특정 문서]
- **결과**: 정합 / 불일치 발견
- **불일치 항목**: (있으면)
  - [문서명]: [문제] → [수정 내용]
- **갱신한 문서**: [목록]
```

## 6. 커밋에 ADR 번호 포함 규칙

- ADR과 직접 관련된 커밋에만 스코프에 ADR 번호 추가
- 형식: `feat(module, ADR-XXX): 설명`
- ADR 없는 일반 커밋은 기존 컨벤션 유지
- 모든 커밋에 무조건 ADR을 붙이지 않음 (의미 희석 방지)

### 예시
```
feat(auth, ADR-007): JWT httpOnly cookie 인증 구현     ← ADR 관련
fix(broker): 토큰 갱신 타이밍 수정                       ← ADR 무관
docs(memory, ADR-020): 에이전트 팀 아키텍처 문서화        ← ADR 관련
test(trading): Kill Switch 테스트 추가                   ← ADR 무관
```

### 트레이드오프
| 방식 | 장점 | 단점 |
|------|------|------|
| **모든 커밋에 ADR** | 완전한 추적 | ADR 없는 커밋에 억지 부여 → 의미 희석 |
| **관련 있을 때만 ADR** (채택) | 추적 가치 유지, 자연스러움 | 일부 누락 가능 |
| **ADR 없이 기존대로** | 단순 | git log에서 설계 추적 불가 |
