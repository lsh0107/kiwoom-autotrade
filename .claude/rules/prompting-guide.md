# Claude Code 프롬프팅 가이드

**사용자와 Claude 모두를 위한 실전 명령 패턴 가이드.**

## 1. 효과적인 명령 패턴

### 구체적으로 지시하기

```
# 나쁜 예
"이거 고쳐줘"
"프론트 만들어줘"

# 좋은 예
"src/auth/login.ts의 validateEmail 함수에서 @가 없는 이메일도 통과하는 버그 수정"
"Next.js로 로그인 페이지 구현. shadcn/ui Button + Input 사용. POST /api/auth/login 호출"
```

### 검증 기준 함께 제공

```
"구현 후 pytest 실행해서 전체 통과 확인"
"ruff check 에러 0개 확인"
"커버리지 85% 이상인지 확인"
```

### 탐색 → 계획 → 실행 순서

```
"먼저 src/trading/ 읽고 현재 주문 플로우 파악한 다음, 취소 주문 기능 추가 계획 세워"
"@src/broker/kiwoom.py 이 파일 분석하고 리팩토링 포인트 정리해줘"
```

## 2. 팀 작업 요청법

```
# 좋은 예 - 역할과 산출물 명확
"팀으로 작업해. backend가 API 만들고, frontend가 UI 만들고, qa가 테스트"
"backend 혼자 버그 수정하고 qa 검증"

# 나쁜 예 - 모호
"다 해줘"
"적당히 팀 나눠서 해"
```

역할 목록은 `.claude/rules/agent-roles.md` 참조.

## 3. 컨텍스트 관리

| 커맨드 | 용도 | 언제 쓰나 |
|--------|------|----------|
| `/plan` | 읽기 전용 분석 | 코드 이해, 설계 검토 |
| `/clear` | 컨텍스트 초기화 | 새 작업 시작 |
| `/compact` | 컨텍스트 압축 | 긴 세션에서 메모리 절약 |
| `/review` | PR 코드 리뷰 | PR 생성 전 |
| `/context` | 컨텍스트 사용량 | 토큰 잔량 확인 |
| `/cost` | 토큰 통계 | 비용 확인 |

- `@파일경로` 로 특정 파일을 직접 참조 가능

## 4. 안티패턴 (하지 말 것)

| 패턴 | 이유 |
|------|------|
| 한 번에 5개+ 기능 요청 | 품질 저하, 컨텍스트 초과 |
| 중간에 맥락 없이 다른 작업 전환 | 혼란, 불완전한 결과 |
| "아까 그거" (구체적 참조 없음) | Claude가 추측하게 됨 |
| 에러 스택 없이 "에러 나" | 디버깅 불가 |
| 확인 없이 "머지해" | Actions 실패 가능 |

## 5. 이 프로젝트 맞춤 팁

### 커밋 전 필수
```
"pre-commit 돌려"          # ruff + mypy + 보안 검사
"커버리지 85% 이상 확인"     # 미만 시 커밋/PR 금지
```

### 트레이딩 기능
```
"모의투자 모드로 테스트"      # is_mock_trading=True 기본값
"주문 안전장치 확인"          # 금액 제한, 횟수 제한, 시간 체크
```

### 작업 마무리
```
"세션 로그 저장"             # sessions/YYYY-MM-DD.md에 기록
"PR 만들어줘"               # → Actions 확인까지 자동
```

### Git 워크플로우
```
"feat/xxx 브랜치에서 작업"    # main/dev 직접 push 금지
"dev로 PR 만들고 체크해"      # gh pr checks 확인 필수
```

## 6. 고급 팁

### 단계적 접근이 필요한 큰 작업
```
"Phase 1: API 엔드포인트만 구현. Phase 2는 다음에"
"먼저 설계 리뷰하고, 승인하면 구현 시작"
```

### 디버깅 요청
```
"이 에러 로그 분석해줘: [스택 트레이스 붙여넣기]"
"pytest -x --tb=short 결과 기반으로 실패 원인 찾아"
```

### 코드 이해
```
"/plan src/trading/order.py의 주문 처리 흐름 설명"
"이 함수의 엣지 케이스 뭐가 있을까?"
```
